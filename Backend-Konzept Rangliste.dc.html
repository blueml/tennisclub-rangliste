Backend-Konzept: Tennisclub-Rangliste

Single-Node-Betrieb im Docker-Container · Speicherung als JSON-Dateien · Backend in Python · Stand 11. Juli 2026

1. Überblick

Die Anwendung läuft als ein Prozess in einem Docker-Container auf einem einzelnen Linux-Host. Es gibt bewusst keine verteilte Architektur und keine Datenbank — alle veränderlichen Daten liegen in JSON-Dateien auf einem gemounteten Docker-Volume. Das hält Betrieb und Backup denkbar einfach und passt zur Größe eines einzelnen Vereins.

Zwei Rollen greifen auf dieselbe Anwendung zu: Admin (verwaltet Spieler:innen, entscheidet Streitfälle) und Spieler:in (fordert heraus, bestätigt Ergebnisse). Die Admin-Zugangsdaten werden separat von den Spielerdaten gepflegt, da sie sich selten ändern und nicht über die App-UI bearbeitbar sind.

Die Baum-/Pyramidenposition jeder Person ergibt sich rein aus ihrem Index im ranking-Array (Zeile/Spalte werden aus der Dreieckszahl-Formel berechnet) — das Array ist die einzige Quelle der Wahrheit für die Rangfolge, damit Gewinn-Rotationen ein simples Array-Splice bleiben.

matches.json

{
  "matches": [
    {
      "id": "m_0007",
      "challengerId": "p_sophie",
      "defenderId": "p_mia",
      "status": "open",
      "confirmations": { "p_sophie": "p_sophie" },
      "createdAt": "2026-07-10T18:04:00Z",
      "resolvedAt": null,
      "resolvedBy": null
    }
  ]
}

status: open · resolved · cancelled. confirmations sammelt pro Spieler-ID die von ihr gemeldete Gewinner-ID; das Match schließt automatisch, sobald beide Einträge vorhanden sind und übereinstimmen. resolvedBy ist null bei beidseitiger Bestätigung, sonst die Admin-ID (Override).

3. Technologie-Stack

  • FastAPI (Python 3.12) als Web-Framework — asynchron, mit Pydantic-Modellen für Player, Match, Admin-Konto und automatischer Request-Validierung.
  • Uvicorn als ASGI-Server im Container.
  • passlib[bcrypt] für Passwort-Hashing, PyJWT für signierte Session-Cookies.
  • Eigene kleine JsonStore-Klasse pro Datei (siehe Abschnitt 4) — keine ORM/DB-Bibliothek nötig.
  • Das bestehende React-Frontend wird als statischer Vite-Build von FastAPI mitausgeliefert (StaticFiles-Mount) — ein Container, ein Deploy.

4. Nebenläufiger Dateizugriff

JSON-Dateien haben kein eingebautes Transaktions- oder Sperrkonzept — das muss die Anwendung selbst sicherstellen, sonst überschreiben sich parallele Schreibzugriffe (z. B. zwei gleichzeitige Ergebnisbestätigungen):

  • Serialisierte Schreibqueue pro Datei im Prozess: jede Änderung an players.json bzw. matches.json läuft durch ein asyncio.Lock je Datei, sodass immer nur ein Schreibvorgang gleichzeitig aktiv ist. Ausreichend, solange nur ein Anwendungsprozess läuft (Single-Node-Vorgabe, ein Uvicorn-Worker).
  • Atomic Write: neuen Inhalt nach players.json.tmp schreiben, dann per rename() über die Zieldatei legen — verhindert kaputte/halb geschriebene Dateien bei einem Absturz mitten im Schreibvorgang.
  • Read-modify-write innerhalb der Queue: jede Operation liest die aktuelle Datei, wendet die Änderung im Speicher an und schreibt sie zurück, bevor die nächste Operation in der Queue startet — das ersetzt row-level Locking einer Datenbank.

5. Authentifizierung

  • Passwörter nie im Klartext speichern — bcrypt-Hash über passlib in beiden JSON-Dateien.
  • Login erzeugt ein signiertes Session-Cookie (JWT via PyJWT, httpOnly + Secure) mit Rolle (admin/player) und bei Spieler:innen der playerId.
  • Server-seitige Middleware prüft bei jedem Request die Rolle: Admin-Endpunkte erfordern role=admin, Herausforderungs-/Bestätigungs-Endpunkte erfordern role=player und mustChangePassword=false.
  • Neu angelegte Spieler:innen starten mit mustChangePassword: true; der Passwortänderungs-Endpunkt ist der einzige, den ein Spieler mit gesetztem Flag aufrufen darf, bevor er/sie sonst etwas tun kann.

7. Serverseitige Regellogik

Die Herausforderungsregeln (links in der gleichen Reihe, eine Reihe höher rechts der eigenen Position) sowie die Gewinn-Rotation (Herausforderer übernimmt die Position, dazwischenliegende Spieler:innen rutschen je eine Position tiefer) müssen serverseitig validiert bzw. angewendet werden — das Frontend darf sie nur spiegeln, damit kein Client durch manipulierte Requests ungültige Matches erzeugt oder Ränge verändert. Sperrzustand ("bereits in einem offenen Match") wird bei jedem POST /api/matches gegen matches.json geprüft.

Ein einzelner Container (FastAPI + Uvicorn) bedient Backend-API und die ausgelieferten Frontend-Dateien. Kein zweiter Datenbank-Service nötig.

9. Backup

Da alle Daten als Dateien im Volume liegen, genügt ein regelmäßiges Kopieren von /data/*.json (z. B. täglicher Cron-Job auf dem Host, der das Volume-Verzeichnis in ein Backup-Ziel sichert). Vor jedem Schreibvorgang sorgt der Atomic-Write-Mechanismus dafür, dass ein Backup nie eine halb geschriebene Datei erwischt.

10. Bekannte Grenzen

  • Skaliert nicht horizontal (mehrere Container-Instanzen/Worker) — bewusst akzeptiert für Single-Node-Betrieb; Uvicorn läuft mit genau einem Worker, damit der asyncio.Lock pro Datei wirksam bleibt.
  • Keine SQL-Abfragen/Reports über historische Matches ohne die gesamte Datei zu laden — bei einem Verein (zweistellige Mitgliederzahl) unkritisch.
  • Wachstum von matches.json: abgeschlossene Matches können nach z. B. 2 Jahren in eine separate matches-archiv.json ausgelagert werden, falls die Datei unhandlich wird.
